countdown game

we're going to create a game about time. It primarily orients around a small number of player characters who can be ordered to move around a graph over time and take actions which may take time to complete to gather materials to produce "utility" before the global timer reaches completion. Time passes continuously, but it can be paused, so gameplay events need to be driven by an update(double delta) function. The graphical style will be minimalistic. The player's ultimate goal in each map is to generate a certain amount of eudaimonia before a global timer runs down.

There are many "may or may not" or "some" choices in this design doc. All of these should be controlled by the Parameters object. Different game levels will have different Parameters. I will set the parameters. You will guess at some reasonable default parametization for the first test, but it's okay if they're imperfect, I will have to tune them.

A common base class is Thing. Things can have positions in the graph. They can have move paths scheduled, represented with a List of graph nodes and a departure times (for now only players will move).

Take a look at the old journeying_game commits for the high level UI logic of the game. So, a view onto the graph/world displays at the top of the screen. It centers on the last selected player. At the bottom of the screen, the available players are shown along the right side of the screen. There may be any number of players between 1 and 5. When a player is selected, their controls and inventory are shown in the left part. A player may or may not have the move action, but initially they all will. The move action is controlled by tapping and dragging in the direction of the wire they want to schedule going along. If no edge's angle is within pi/2 of the dragged move, don't move. Actions also include the functionality of the local graph node that they're currently in. Traders, train stations, and trees. Trader actions initiate that trader's trade. Train station actions allow scheduling a move of the train in the same way moves are scheduled for players. Tapping trees takes their current items. Players can only hold 5 items (though this should be a parameter I guess).

items
	there's a spectrum between basic items and higher items, evenly spread, with some clustering towards the higher end. Some traders trade basic items for other basic items. Most traders trade multiple basic items or higher items for other higher items.
	utility is generally one of the most high items.
	basic items are circles, vertical pills, or diamonds, each one can come in different colors. Colors are parameters, and there can be any number of them, but for now the colors will be yellow-orange, cyan, red-magenta, very dark grey, pale grey, and sage.
	Higher items (except utility) are these shapes combined in various arangements. They should be specified as operator trees, such as Horiz(Nest(Diamond(Sage), Pill(Cyan)), Diamond(Red)). This is just a visual mnemonic, to make it easier to learn how some traders will make them by combining those inputs in that order, but some other traders might make them in a totally different way. No trader should nest an operator inside of the same operator, as this would produce visually ambiguous icons. If an item is nested in an item of the same color, the widget must be passed outlineOn = true.
	Eudaimonia looks like a blood-red heart.

things in the world:
	some nodes are "trains" that connect to a station node and can be moved to another station node when triggered, the amount of time they take is shown, they often travel along a shortcut that nothing else can travel along
		stations

			render: 's' then small Icons.train

level generation:
	use a seed
	initialize the graph as a grid of gridSizeN x gridSizeN, nodes are initially 3.5 units apart.
	apply ciel(gridSizeN*(gridSizeDistortionCountStartp + rng*gridSizeDistortionCountVariancep)) distortions to the grid
		distortions consist of for either horizontal or vertical, picking a position partway between the two sides of the grid, then pushing the nodes on either side of the line apart by rng*gridSizeDistortionp
	reposition so that map is centered on (0,0), note map bounds
	select each line or column and
		transaction, rattempt 8 times at max, otherwise skip
			if rng > lineRemovalProb
				for each node in the line, delete (along with its edges) if rng > pointRemovalProb.
			if the graph is no longer fully connected, retry the transaction
	for each edge with length > 3
		if rng > middleNodeProb
			split the edge at (where span = max(0, edgeLength - splitNodeMinDistance * 2)), edgeLength/2 + rng * span - span/2, a new node goes there
	facilities should be generated at a fixed distribution, eg, parameters determine that there will be a certain number of storages, traders, etc, in the world. They're then randomly put into buckets according to a fixed density distribution. Then the buckets are randomly strewn around the world. Trains can contain facilities just like any other node!
	nTrainSchedules are generated randomly, which can be in the form of Never (which will only move when the player moves it), OneWay (which returns to its initial station two seconds after arrival), or Cycle(seconds) (which also can't be controlled) (seconds should be generated to one decimal place). Trains are assigned these on scheduleDistribution.
